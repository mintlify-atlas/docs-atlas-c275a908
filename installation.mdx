## Requirements

- **Python 3.9 or later** — VITACORE uses type hints and f-strings that require Python 3.9+.
- **pip** — comes with all standard Python distributions.
- A **Supabase project** with Auth enabled and your tables provisioned.

## Set up a virtual environment

Using a virtual environment isolates VITACORE's dependencies from your system Python.

```bash
python -m venv .venv
```

Activate the environment:

```bash
# macOS / Linux
source .venv/bin/activate

# Windows (Command Prompt)
.venv\Scripts\activate.bat

# Windows (PowerShell)
.venv\Scripts\Activate.ps1
```

## Install dependencies

```bash
pip install -r requirements.txt
```

The `requirements.txt` contains:

```text
flask
supabase
python-dotenv
```

| Package | Role |
|---|---|
| `flask` | HTTP routing, session management, and Jinja2 template rendering |
| `supabase` | Official Supabase Python client for database queries and Auth |
| `python-dotenv` | Reads key-value pairs from `.env` into `os.environ` at startup |

## Environment variables

VITACORE reads all configuration from environment variables via `python-dotenv`. Create a `.env` file in the project root before starting the server.

<Warning>
  Do not use the default value `dev-secret` for `FLASK_SECRET_KEY` in any environment where real patient data is stored. Generate a strong random key and keep it secret.
</Warning>

```bash
FLASK_SECRET_KEY=replace-with-a-long-random-string
SUPABASE_URL=https://your-project-ref.supabase.co
SUPABASE_ANON_KEY=your-anon-public-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-secret-key
SUPABASE_TABLE_HC_PROFESIONALES=hc_profesionales
```

### Variable reference

<ParamField path="FLASK_SECRET_KEY" type="string" required>
  Used by Flask to sign session cookies and CSRF tokens. If not set, the application falls back to the insecure default `dev-secret`. Generate a suitable value with `python -c "import secrets; print(secrets.token_hex(32))"`.
</ParamField>

<ParamField path="SUPABASE_URL" type="string" required>
  The base URL of your Supabase project, e.g. `https://abcdefghijkl.supabase.co`. Find it in your Supabase dashboard under **Settings → API → Project URL**.
</ParamField>

<ParamField path="SUPABASE_ANON_KEY" type="string" required>
  The public anonymous key for your Supabase project. Used by `get_supabase_public()` for operations that run under the authenticated user's Row Level Security context, including the login flow.
</ParamField>

<ParamField path="SUPABASE_SERVICE_ROLE_KEY" type="string" required>
  The service role secret key for your Supabase project. Used by `get_supabase_admin()` for privileged operations such as user and role management. **Never expose this key to the browser or commit it to source control.**
</ParamField>

<ParamField path="SUPABASE_TABLE_HC_PROFESIONALES" type="string" default="hc_profesionales">
  The name of the Supabase table that stores professional (user) profiles. Defaults to `hc_profesionales` if not set. Override this if your table uses a different name.
</ParamField>

## Running in development mode

The `app.py` entry point starts Flask with `debug=True`:

```python
if __name__ == "__main__":
    app.run(debug=True)
```

Start the server:

```bash
python app.py
```

Flask starts on `http://127.0.0.1:5000` with the reloader and debugger enabled. The debugger exposes an interactive console in the browser on unhandled exceptions — use this only on a trusted local machine.

<Info>
  The development server is single-threaded and not suitable for concurrent users. Use a production WSGI server such as Gunicorn for any shared or public deployment.
</Info>

## Running in production

Disable debug mode and serve with a production WSGI server.

```bash
pip install gunicorn
gunicorn -w 4 -b 0.0.0.0:8000 app:app
```

Set `FLASK_SECRET_KEY` to a strong secret and ensure your `.env` file (or environment variables injected by your hosting platform) are in place before starting the server.

<Warning>
  Always run VITACORE behind HTTPS in production. Patient health data is sensitive and must be encrypted in transit.
</Warning>

## Local data files

VITACORE includes a `data/` directory containing JSON files used as fallback or seed data when certain lookup tables are not yet populated in Supabase. If you encounter missing dropdown options (document types, countries, etc.) on first run, check that your Supabase tables are seeded or that the relevant JSON files are present in `data/`.
